// Detection Pipeline

Phishing Detection Workflow

The detailed, ten-stage process our ML engine uses — from URL ingestion to final threat classification and action enforcement.

Step 01
Client Request

The client — either the web dashboard or browser extension — sends a POST /check-url request with a URL and mode flag.

Action: Validate input · Normalise URL scheme · Extract mode
📡
Mode Flags

viewer — web dashboard, analysis only, no blocking enforced.

protect — browser extension, full block/warn/allow enforcement.

Database Lookup

The domain cache avoids redundant ML computation for known-good or known-bad domains. Whitelist hits return instantly with score 0.

🗄️
Step 02
Domain Extraction & DB Lookup

The registered domain is extracted via tldextract. The database is checked for a whitelist or blacklist entry before any ML work is done.

Action: Extract domain · Query SQLite · Return stored result on hit
Step 03
Stage 1 — Feature Extraction

31 URL-level features are extracted in under 1ms: length, character counts, entropy, subdomain depth, HTTPS flag, brand similarity, keyword count, suspicious TLD.

Goal: Build the Stage 1 feature vector for ML input
🔬
Key Features

url_length · domain_entropy · brand_in_url · keyword_count · suspicious_tld · has_ip · subdomain_depth · digit_ratio

Random Forest

200 decision trees trained on balanced real + synthetic data. StandardScaler normalises features before prediction. Output is P(phishing) in [0,1].

🤖
Step 04
Stage 1 ML Prediction (S1)

The Random Forest classifier predicts P(phishing). The output is scaled to a 0–100 risk score. This is the fast first-pass filter.

Output: S1 score 0–100 · Decision threshold applied
⚡ Stage 1 Decision Gate

Three routing paths based on S1 score — only the grey zone proceeds to Stage 2

S1 ≥ 90
→ Phishing · Block
65 ≤ S1 < 90
→ Stage 2 Analysis
S1 < 65
→ Safe · Allow
Step 05
Stage 2 — Deep Feature Extraction

16 additional behavioral features are extracted for grey-zone URLs: login form detection, external form action, base64 in query, punycode domains, redirect chains, unicode tricks.

Goal: Expose obfuscation and behavioral phishing signals
⚗️
Stage 2 Signals

has_login_form · external_form_action · base64_in_query · encoded_domain · unicode_in_domain · fake_https_prefix · multiple_http · long_subdomain

Combined Input

Stage 2 model receives all 47 features (31 S1 + 16 S2). This gives a richer picture of behavioral intent beyond URL structure alone.

🧠
Step 06
Stage 2 ML Prediction (S2)

A deeper Random Forest (300 trees) runs on the combined 47-feature vector. Output is a refined risk score with higher confidence on ambiguous URLs.

Output: S2 score 0–100 · Final classification ready
🚨 Stage 2 Decision Gate

Final classification for grey-zone URLs

S2 ≥ 80
→ Phishing · Auto-Blacklist
S2 < 80
→ Suspicious · Warn
Step 07
Mode-Based Action Layer

The final risk score is converted to an action based on the request mode. Viewer mode always returns "none". Protect mode enforces block/warn/allow.

Action: Map status × mode → block | warn | allow | none
⚙️
Action Map

phishing + protect → block
suspicious + protect → warn
safe + protect → allow
any + viewer → none

Auto-Blacklist

Domains scoring ≥ 90 are automatically written to the blacklist. Future requests to the same domain return instantly — no ML required.

💾
Step 08
Persistence & Auto-Blacklisting

Every scan is written to the SQLite audit log. High-risk domains are upserted into the blacklist table. The URL cache is invalidated for consistency.

Action: INSERT scan · UPSERT domain if score ≥ 90 · Invalidate cache
Step 09
Response Serialisation

The Pydantic CheckURLResponse model validates and serialises the output. CORS headers are attached for browser extension cross-origin access.

Output: JSON · risk_score · status · action · reasons[ ]
📤
Response Schema

risk_score: 0–100
status: safe|suspicious|phishing
action: allow|warn|block|none
reasons: string[]

Extension Enforcement

The extension reads the action field. block → redirect to warning.html. warn → inject banner. allow → proceed silently.

🔒
Step 10
Client-Side Enforcement

The web dashboard displays the result card with colour-coded risk. The browser extension enforces the action — blocking, warning, or allowing navigation.

End result: User is protected or informed in real time
